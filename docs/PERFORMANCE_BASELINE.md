# Sound 性能基线

最后更新：2026-07-13

## 目的与边界

这份基线固定资料库 SQLite 读写、模型映射与搜索管线，为 SND-502 后续优化提供
可重复对照。基准使用合成资料：每张专辑 10 首歌，10% 的歌曲带 5 行歌词，搜索
命中歌曲艺人与流派并按相关度排序，最多返回 200 首。

当前结果是 **Debug/JIT 微基准**，不等于真实用户帧率：

- 包含真实 SQLite 临时文件的整批扫描写入、重开读取、资料库记录到界面模型的
  映射、搜索索引、后台 isolate 查询，以及 10,000 首合成长列表的可视区域构建；
- 不包含真实音频文件枚举与元数据解析、首帧、真实封面解码、GPU 滚动、平台
  输入法和网络；
- RSS 是测试进程的同机相对参考，不能与其他设备直接比较；
- 后续仍需 macOS/Android Profile 构建与 iPhone/iPad 真机数据。

## 固定环境

- 设备：MacBook Pro（Apple M5，10 核，32 GB）
- 系统：macOS 26.5.1
- Flutter：3.44.6 stable
- Dart：3.12.2
- 运行命令：

```sh
flutter test benchmark/library_performance_test.dart --reporter expanded
flutter test benchmark/library_database_performance_test.dart --reporter expanded
flutter test benchmark/library_scroll_performance_test.dart --reporter expanded
```

纯内存管线先预热，再取 5 次异步或 7 次同步样本的中位数。SQLite 文件基准分别
运行一次首次扫描和同内容重扫，并在关闭重开后分段读取。运行时应关闭其他高负载
任务，SQLite 数值应至少复跑两次确认范围。

## 已确认并处理的瓶颈

### 歌词 N+1 查询

原实现刷新资料库时对每首歌调用一次 `getLyrics`，10,000 首会产生 10,000 次
数据库查询。现在统一使用一次 `getAllLyrics` 批量读取并按歌曲分组；控制器测试明确
约束批量查询次数，逐曲查询必须为零。扁平歌曲列表也只在资料库刷新时生成一次，
界面读取不再重复展开全部专辑。

### 连续输入重复规范化

原搜索每次请求都会重新规范化五个字段。两次稳定复跑中，10,000 首后台搜索 P50
为 41.21–41.36ms。现在在资料库变化时一次性建立规范化索引，后台搜索降至
2.68–2.98ms；代价是一次性索引约 37ms，测试进程 RSS 增加约 9–10MiB。

当前取舍优先保证连续输入响应。一次性索引仍可能造成约两帧的启动/重扫抖动，
后续应评估把大库索引也移入长期工作 isolate，或改用 SQLite FTS，同时记录真实
设备内存，避免以过多常驻字符串换取速度。

### 整库重扫的逐行冲突更新

最初的真实 SQLite 基准中，10,000 首同内容重扫耗时 1,220.93ms。主要开销来自
大规模 `NOT IN` 参数列表，以及在完整快照上逐条执行冲突更新。扫描结果本来就是
单一来源的完整快照，因此现在在同一个事务中清空该来源的歌曲、专辑和艺人，再用
Drift 批量写入新快照；任意写入失败仍会回滚到旧资料库。

四次稳定复跑中，10,000 首同内容重扫降至 122.18–134.26ms，较原始结果缩短约
89%；首次写入为 130.41–148.60ms。空快照清理、旧 ID 收敛、失败回滚、本地重扫
和 WebDAV 重扫均有自动化覆盖。未来加入收藏、历史等用户数据表时，不能让这些
数据依赖会随整批重建级联删除的目录行；SND-506 增量扫描也会重新评估这个取舍。

### 原始封面解码与不定高歌曲列表

原封面组件无论显示尺寸多小，都直接使用原始 `FileImage`/`NetworkImage`；48px
歌曲缩略图因此也可能把 1000×1000 甚至更大的图片完整解码到 ImageCache。现在
组件根据布局尺寸和设备像素比请求解码，并向上归入 64 物理像素档位，接近尺寸的
缩略图可共享缓存；小于 96 逻辑像素时也不再绘制 28px 模糊阴影。以 @2x 的
48–52px 缩略图为例，两者共用 128×128 档位，单张解码上限约 64KiB；相比
1000×1000 RGBA 的约 3.8MiB，理论下降约 98%。非方形封面使用保持比例的缩放策略。

歌曲列表原先使用不定高 `SliverList`。10,000 首基准做大距离定位时，框架需要逐步
估算行高，超过 50 秒仍未完成。现在使用真实歌曲行作为一次性高度原型，并限制标题
和副标题为单行；既保留主题和字号决定的实际高度，也让完整滚动范围可直接计算。
优化后三次复跑的 40 次跨库位置抽样稳定在 1.19–1.30 秒，存活封面最多 24 个。
