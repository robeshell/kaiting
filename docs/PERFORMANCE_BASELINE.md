# Sound 性能基线

最后更新：2026-07-13

## 目的与边界

这份基线先固定资料库模型映射与搜索管线，为 SND-502 后续优化提供可重复对照。
基准使用合成资料：每张专辑 10 首歌，10% 的歌曲带 5 行歌词，搜索命中歌曲艺人
与流派并按相关度排序，最多返回 200 首。

当前结果是 **Debug/JIT 微基准**，不等于真实用户帧率：

- 包含资料库记录到界面模型的映射、搜索索引和后台 isolate 查询；
- 不包含 SQLite 文件读取、首帧、真实封面解码、GPU 滚动、平台输入法和网络；
- RSS 是测试进程的同机相对参考，不能与其他设备直接比较；
- 后续仍需 macOS/Android Profile 构建与 iPhone/iPad 真机数据。

## 固定环境

- 设备：MacBook Pro（Apple M5，10 核，32 GB）
- 系统：macOS 26.5.1
- Flutter：3.44.6 stable
- Dart：3.12.2
- 运行命令：

```sh
flutter test benchmark/library_performance_test.dart --reporter expanded
```

每项先预热，再取 5 次异步或 7 次同步样本的中位数。运行时应关闭其他高负载任务。

## 2026-07-13 基线

| 规模 | 资料库刷新 P50 | 一次性搜索索引 P50 | 后台搜索 P50 | RSS 参考 |
| --- | ---: | ---: | ---: | ---: |
| 1,000 首 / 100 专辑 | 1.19–1.20 ms | 3.75–3.90 ms | 0.49–0.50 ms | 约 169 MiB |
| 10,000 首 / 1,000 专辑 | 2.94–3.52 ms | 37.08–37.42 ms | 2.68–2.98 ms | 约 187 MiB |

本机手动回归预算：10,000 首资料库刷新低于 50ms、搜索索引低于 80ms、后台搜索
低于 15ms；1,000 首分别低于 15ms、15ms 和 5ms。预算写入独立基准测试，
不会拖慢日常测试套件。

## 已确认并处理的瓶颈

### 歌词 N+1 查询

原实现刷新资料库时对每首歌调用一次 `getLyrics`，10,000 首会产生 10,000 次
数据库查询。现在统一使用一次 `getAllLyrics` 批量读取并按歌曲分组；控制器测试明确
约束批量查询次数，逐曲查询必须为零。扁平歌曲列表也只在资料库刷新时生成一次，
界面读取不再重复展开全部专辑。

### 连续输入重复规范化

原搜索每次请求都会重新规范化五个字段。两次稳定复跑中，10,000 首后台搜索 P50
为 41.21–41.36ms。现在在资料库变化时一次性建立规范化索引，后台搜索降至
2.68–2.98ms；代价是一次性索引约 37ms，测试进程 RSS 增加约 9–10MiB。

当前取舍优先保证连续输入响应。一次性索引仍可能造成约两帧的启动/重扫抖动，
后续应评估把大库索引也移入长期工作 isolate，或改用 SQLite FTS，同时记录真实
设备内存，避免以过多常驻字符串换取速度。

## 下一轮测量

- 真实 SQLite 文件：打开、查询、批量歌词读取和资料库映射分段耗时；
- macOS/Android Profile：冷启动首帧、搜索连续输入和长列表逐帧数据；
- 1,000/10,000 首真实封面：解码时间、ImageCache、峰值 RSS 和滚动回收；
- WebDAV 大库：索引恢复、远程封面、网络流量与缓存命中；
- iPhone/iPad 真机：输入法、分屏、内存警告和后台恢复。
